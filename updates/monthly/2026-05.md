# May 2026 Research Summary

May was the month when AI tooling stopped being about capability demos and started being about production infrastructure. Cursor shipped persistent cloud dev environments for agents, Microsoft shipped Agent 365 as a control plane for enterprise agent governance, and Notion proved that agents embedded in existing workflows (like private Slack channels) beat standalone agent products every time. The signal underlying everything: the question isn't "what can AI do?" anymore — it's "how do we actually deploy this at scale without losing control?"

The secondary pattern running underneath this month is the economics reset. Harrison Chase said out loud what everyone building agents is thinking: "the cost of closed models is too high." LangChain is betting on open-source models for production agent work. Even Lenny published a breakdown explaining why SaaS freemium playbooks fail for AI (inference costs are real, not theoretical). For PMs, this means the unit economics conversation is no longer optional — it's foundational to whether your agent-powered feature is actually viable.

## What Matters

- **Enterprise AI adoption now requires governance infrastructure, not just capability.** Cursor's always-on security agents (gated to Teams/Enterprise), Microsoft Agent 365's centralized identity and audit trails, and Notion's audit logs for Custom Agents all shipped this month with the same underlying message: enterprises won't deploy agents without seeing what they do, controlling what models they use, and tracking the spend. This shifted the competitive surface for AI tools from "which has the best model?" to "which gives me the controls my security team needs?" Cursor's security review agents are the clearest example — they're not solving a developer problem, they're solving a procurement problem, which is why they're plan-gated to force upgrades.

- **The PM-as-builder pattern is now validated at production scale.** Owen Williams at Stripe vibe coded Protodash in days and it transformed how design reviews work at one of the most sophisticated product companies. Teresa Torres (the person who invented Opportunity Solution Trees) ran an engineering sprint to build AI-generated OSTs. These aren't novelties or weekend projects — they're PMs shipping real product leverage using AI tools. The pattern is: PM has domain expertise + AI tool for execution = capability shift for the entire team. If your PM skill set is "writing detailed requirements," you're increasingly in trouble. If it's "building leverage for your own workflows," you're just getting started.

- **Agents embedded in existing context (workspaces, Slack channels, dev environments) outperform standalone agents every time.** Notion's Custom Agents work better inside existing Notion databases than as a blank-slate tool. Cursor's cloud dev agents work because they have the actual repo context, not sandboxed pretend context. Stripe's agent payment layer works because it's wired into existing Link infrastructure, not a separate system. The moat isn't building a better agent — it's having the context the agent needs to actually do something useful. This is directly relevant to any PM thinking about where to place agentic features: embedding beats bolting-on.

## Essential Resources

1. **Cursor** — [Persistent cloud development environments for agents](https://cursor.com/changelog/05-13-26) — Agents can now operate inside configured environments with repo access, credentials, and build systems, fundamentally changing the ROI on cloud agents for completing real engineering work.

2. **Notion** — [What We Learned During the Custom Agents Beta](https://www.notion.com/blog/what-we-learned-during-the-custom-agents-beta) — First-party post-mortem on autonomous agents inside a productivity product; the transparency finding (users don't trust agents they can't audit, even correct ones) is directly actionable for any PM building agentic features.

3. **Lenny Rachitsky** — [Why SaaS Freemium Playbooks Don't Work in AI](https://www.lennysnewsletter.com/p/why-saas-freemium-playbooks-dont) — Breaks down the cost structure difference between AI and SaaS; includes the "capability-limited free tier" pattern that actually works for inference-based products.
